## Gap unit outline

**Topic:** Tradeoff-driven design practice — how to structure design decisions and present them clearly.

**What it covers:**
- Naming the tradeoffs before picking an option
- Structured whiteboard / doc format for design sessions
- Interview timing and communication strategies
- Using worked examples to build pattern fluency

**Why it matters:** Being able to articulate tradeoffs — not just pick the "right" answer — is what separates strong system designers. Every design choice omits alternatives; stating why matters.

**In BRIEFR:** BRIEFR's own decisions (monolith over microservices, SQLite over Postgres at small scale, in-process cache over Redis, single-tenant over SaaS) are ready-made worked tradeoff examples.

**Why BRIEFR skips this as a topic:** Tradeoff reasoning is a meta-skill across all units, not a separate implementation feature.

**When you would need it:** Design interviews; architecture review boards; documenting ADRs for a team.
