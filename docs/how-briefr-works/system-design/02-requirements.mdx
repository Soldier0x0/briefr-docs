# Unit 2: Requirements & framing

<CoverageBadge coverage="partial" />

Every design starts before the first component is chosen. Before deciding between SQL and NoSQL, before picking a cache strategy, before drawing a box labeled "scheduler" — a good designer pins down what the system actually needs to do and what it explicitly does not need to do. BRIEFR's product brief is unusually explicit about both sides of that line.

## BRIEFR's locked requirements

BRIEFR is a **self-hosted CVE intelligence portal** for a **small security team** with these hard constraints:

**Single deployment model:** One box, one operator, one organization. No multi-tenancy, no SaaS infrastructure, no shared database. The operator installs and owns the full stack.

**Single writer:** One backend process writes to the database. The scheduler, the enrichment jobs, and any admin-triggered refreshes all run inside that one process. This is enforced by in-process locking (`scheduler_locks.py`) and by deployment guidance that restricts the scheduler to one worker instance.

**Bounded corpus:** The data is the publicly available CVE feeds — NVD, KEV, EPSS, MITRE ATT&CK, cvelistV5, Vulnrichment, and correlated threat intelligence. The corpus grows, but it is not user-generated at arbitrary volume.

**Small concurrent user base:** Single-digit simultaneous analysts. Throughput is not the primary design concern. Accuracy and reliability of intelligence data are.

**No external API consumers:** BRIEFR's API is consumed by its own bundled frontend and operator scripts. There are no external API partners, no versioning SLAs, and no mobile clients.

**Explicit non-goals (by product design):**

- Multi-tenant SaaS (see [Unit 14](/docs/how-briefr-works/system-design/14-multi-tenancy))
- Active-active multi-region failover
- Real-time streaming ingest at high volume
- External identity provider integration (SAML/OIDC)
- General-purpose distributed systems patterns (consensus, leader election — see [Unit 7](/docs/how-briefr-works/system-design/07-distributed))

## How fixed constraints simplify design

The value of explicit non-goals is what they let you skip. Because BRIEFR is a single-writer system, there is no need for optimistic concurrency control, write-conflict resolution, or distributed lock managers. Because it is single-tenant, there is no row-level tenant isolation, no billing-aware resource quotas, and no noisy-neighbor problem. Because the corpus is bounded, there is no need for horizontal shard planning or index rotation at scale.

These are not omissions — they are savings. Every distributed-systems mechanism that BRIEFR does not implement is an operational burden that its operator does not carry. The design earns simplicity by being honest about scope.

This framing — "what does the product explicitly not do, and what does that let us skip?" — is one of the most useful moves in any system-design conversation.
