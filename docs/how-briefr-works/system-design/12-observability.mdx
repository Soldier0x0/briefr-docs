# Unit 12: Observability & operations

<CoverageBadge coverage="partial" />

A system that fails silently is worse than a system that fails loudly. BRIEFR's operations design is built on that principle: health state is exposed at a structured endpoint, significant events push notifications rather than requiring polling, backups announce their own staleness, and admin operations are surfaced in a single panel rather than scattered across config files.

The full engineering detail lives in [Webhooks and operations](/docs/how-briefr-works/how-its-built/webhooks-ops).

## Health-check endpoint

`GET /api/health` is BRIEFR's primary observability surface. It reports:

- **Source health** — per-feed circuit-breaker state: `last_success`, `last_failure`, `last_error`, `consecutive_failures`, `circuit_open` for each configured intelligence source.
- **Database status** — whether the primary database connection is alive and responding.
- **Scheduler status** — whether the APScheduler instance is running and which jobs are currently executing.
- **System resources** — disk usage, memory, and CPU metrics sampled by the resource metrics background job.

This endpoint is the first place an operator checks when something seems wrong. A glance at `feeds.sources` tells you which sources are healthy, which have tripped their circuit breakers, and what errors they last produced. No log parsing required.

## Webhook notifications and the operator alert model

BRIEFR pushes alerts to operator-configured destinations rather than requiring the operator to poll:

- **KEV stack hit:** A CVE on the team stack just entered CISA's Known Exploited Vulnerabilities catalog.
- **KEV backlog:** A new gap appeared in the detection backlog.
- **Watchlist IOC match:** An IOC from the watchlist appeared in fresh threat feeds.
- **Backup dead-man:** The most recent backup archive is older than the configured threshold — a signal that backups have stopped running.

Supported destinations: Discord webhooks, Telegram bots, any HTTPS endpoint accepting a JSON POST. SSRF validation runs on every destination URL before delivery.

The alert model is push rather than pull: the operator does not need to check BRIEFR to know that something significant happened. This is especially important for a self-hosted tool that runs unattended — the analyst is not watching a dashboard; the dashboard calls the analyst.

## Operator notification center

The admin interface includes a notification center that aggregates:

- **Job errors** from background jobs and enrichment tasks.
- **API key health failures** — when a provider key is expired, rate-exhausted, or rejected.
- **Audit events** — session revocations, admin role changes, and other security-relevant operations.

This gives operators a single surface for reviewing what has happened in the background without requiring access to raw application logs.

## Backup and restore

BRIEFR's backup system (`backup/manager.py`) runs on a configurable schedule:

- **Postgres:** uses `pg_dump` to produce a consistent snapshot. Backups are optionally encrypted at rest.
- **SQLite:** copies the database file at a quiescent moment.

Archives are written to a configurable directory. The backup dead-man alert fires if no fresh archive appears within the threshold period. Restore is handled by `ensure_db_or_restore()`, which is called at startup and attempts restoration from the most recent archive if the database is absent or corrupted.

## Admin operations surface

The admin panel exposes:

- **Scheduler control:** pause, resume, and manually trigger individual jobs.
- **Job history:** last run time, duration, and error state for each job.
- **Outbound job queue:** status of Procrastinate tasks when the durable queue is enabled.
- **API key health:** per-provider health ping results and last-seen error.
- **Session management:** active sessions enumerable and revocable per user.

<AtEnterpriseScale>
  At enterprise scale, observability is a full discipline: distributed tracing (OpenTelemetry spans propagated across service boundaries), centralized metrics (Prometheus + Grafana or a managed equivalent), structured log aggregation (Datadog, Splunk, Elastic), and SLO dashboards with error-budget burn-rate alerts. Feature flag platforms (LaunchDarkly, Unleash) let operators gradually roll out changes and kill-switch features in production without deploys. BRIEFR's model — one health endpoint, push webhook alerts, a local notification center — is the correct simplification for a single-box self-hosted tool where the "platform team" is one operator.
</AtEnterpriseScale>
