# Unit 1: Foundations

<CoverageBadge coverage="partial" />

System design rests on a small vocabulary that appears in every design conversation: latency, throughput, availability, reliability, and the trade-offs between them. BRIEFR makes concrete choices along each of these dimensions — choices that are visible in the codebase and observable in a running instance.

## Latency and availability in BRIEFR

**Latency** in BRIEFR's context is primarily analyst-facing: how long does a CVE detail page take to load? How fast does a search return results? The architectural answer is in [Unit 3 (architecture)](/docs/how-briefr-works/system-design/03-architecture): the scheduler pre-fetches and stores all intelligence locally, so the request path reads from a local database rather than calling upstream APIs. A CVE page load is a few SQL queries against a local index, not an outbound network call. This keeps p50 latency in the tens of milliseconds and makes it stable across upstream availability.

**Availability** in BRIEFR's context means: does the analyst UI stay functional when an external feed source goes down? The answer is yes, by design. The scheduler's circuit-breaker model ([Resilience](/docs/how-briefr-works/how-its-built/resilience)) handles source failures in the background. When a source trips its circuit breaker, that source's ingestion pauses; the analyst UI continues reading from the last-stored data. The trade-off is staleness, not downtime. BRIEFR accepts brief feed staleness in exchange for UI availability. This is a deliberate availability-over-consistency choice at the single-box level.

**Soft-fail under pressure** appears in rate limiting. When a caller exceeds BRIEFR's local token-bucket limits (`RATE_LIMIT_IOC_PER_MINUTE`, `RATE_LIMIT_REFRESH_PER_MINUTE`), the API returns `429 Too Many Requests` with a `Retry-After` header rather than queuing requests or blocking. This keeps the system responsive under load at the cost of rejecting excess requests. See [Resilience](/docs/how-briefr-works/how-its-built/resilience) for the full rate-limit model.

**Single-box ceiling.** BRIEFR's availability model rests on one physical or virtual machine. If that machine goes down, BRIEFR goes down. This is not a gap — it is a product constraint. The backup and restore procedure ([Webhooks and operations](/docs/how-briefr-works/how-its-built/webhooks-ops)) minimizes recovery time after a host failure, but it does not provide zero-downtime failover. Operators who need that guarantee need infrastructure outside BRIEFR's scope.

<AtEnterpriseScale>
  At enterprise scale, latency and availability are managed with explicit SLOs — targets like "p99 API response under 200ms" or "99.9% monthly uptime." Error budgets govern how much availability can be spent on risky deployments. Multi-region active-passive or active-active failover removes single-host availability ceilings. BRIEFR's model (local reads, soft-fail, single-box) is the correct simplification for its deployment target. The vocabulary below is what you need when those ceilings matter.

## The external fundamentals

The sections above ground the foundation vocabulary in a real system. The following gap block covers the parts BRIEFR does not implement: formal SLO/SLI/SLA framing and structured back-of-envelope estimation.
