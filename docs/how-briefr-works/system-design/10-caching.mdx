# Unit 10: Caching & performance

<CoverageBadge coverage="partial" />

Caching accelerates reads by storing results that are expensive to recompute. BRIEFR's caching strategy is deliberately simple: two database-backed cache tables hold the expensive results — feed snapshots and IOC enrichment — and the request path reads from those tables rather than calling external APIs. No separate cache process, no distributed cache tier, no cache invalidation protocol.

## `feed_cache`: snapshot results at configured TTL

`feed_cache` stores the last-fetched snapshot for each intelligence feed that BRIEFR polls on a high frequency. Each row is keyed by a source identifier and carries a TTL column that determines how long the cached value is considered fresh.

The cache's write path is the scheduler: on each successful feed pull, the scheduler writes or updates the cache row. The read path is the API route that serves feed data to the analyst UI. When the cached row is fresh (within TTL), the route returns the stored result without calling the external API. When the row is stale or absent, the scheduler's next run refreshes it — but the API can still serve the stale result rather than blocking on an external call.

This design means feed freshness is decoupled from request latency. An analyst loading a feed page never waits for an external API call. The trade-off is that stale data may be shown briefly when a feed is between refresh cycles.

## `ioc_cache`: enrichment results with bounded TTL

`ioc_cache` stores IOC enrichment results from VirusTotal, AbuseIPDB, GreyNoise, CIRCL, and ThreatFox. Enrichment is the most expensive operation in BRIEFR's request path: a cache miss on `POST /api/ioc/lookup` triggers an external API call (or several, for multi-provider enrichment), and those calls consume paid quota.

Cache rows are keyed by IOC value (the IP, hash, or domain being looked up). On a cache hit, the enrichment result is returned from the database with no external call. On a cache miss, the provider is called, the result is stored, and subsequent lookups for the same IOC are served from cache for the TTL duration.

**CIRCL miss caching:** Failed or empty CIRCL lookups — where the provider returns no result — are also cached for 24 hours using `circl_miss:*` keys. This prevents the same missing IOC from re-querying a rate-limited or empty endpoint on every enrichment cycle.

**Retention sweeps:** The scheduler runs `run_retention_cleanup()` on a configurable schedule, deleting cache rows past their physical retention window. This keeps both cache tables from growing unbounded over time.

## In-process vs shared cache trade-offs at single-box scale

BRIEFR's caches live in the database, not in a separate cache process or in-memory store. This is a deliberate choice:

**Persistence across restarts.** Database-backed cache entries survive process restarts. An in-memory cache (a Python dict, a local LRU cache) would be empty after every deploy. For a tool where an analyst might restart BRIEFR and immediately look up IOCs they looked up yesterday, database persistence avoids an unnecessary cold start.

**Simplicity of invalidation.** BRIEFR's cache invalidation strategy is TTL expiry plus retention sweeps — no explicit invalidation events, no cache coherence protocol. This works because the cached data (enrichment results, feed snapshots) changes at the source rather than due to BRIEFR's own write operations. There is no "write-through" case where BRIEFR's own operations must invalidate a cache entry.

**No stampede protection needed.** At BRIEFR's scale (single-digit concurrent users), thundering-herd / cache stampede scenarios — where a large number of concurrent requests all miss the cache simultaneously and trigger parallel external calls — do not arise in practice. Rate limiting on the IOC lookup route (30 req/min/IP) provides an additional bound.

For the embedding cache that backs semantic search, see [Search and embeddings](/docs/how-briefr-works/how-its-built/search-embeddings): embedding vectors are stored in the `embeddings` table and re-generated only when the source content hash changes, which is a form of change-aware invalidation.

<AtEnterpriseScale>
  At enterprise scale, caching operates at multiple layers simultaneously: CDN edge nodes cache static assets and API responses near the user; a distributed in-memory cache (Redis, Memcached) holds hot database results shared across many API workers; application-level caching handles per-request memoization. Cache stampede (thundering herd) becomes a real problem when hundreds of workers simultaneously miss the same cache key after TTL expiry — probabilistic early expiration and request coalescing (only one worker fetches, the rest wait) are standard mitigations. Tail latency at mega-scale is driven by cache misses in the p99+ percentiles, leading to hedged requests, background refresh, and careful cache sizing analysis.
</AtEnterpriseScale>
