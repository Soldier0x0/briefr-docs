# Unit 14: Multi-tenancy & SaaS

<CoverageBadge coverage="gap" />

BRIEFR is deployed once per organization, on infrastructure the operator controls. The multi-tenancy and SaaS architecture patterns that apply when a vendor hosts the same service for many independent customers are deliberately out of scope. This unit maps the topic and explains the trade-off BRIEFR makes by choosing a self-hosted, single-tenant model.

  whyItMatters="SaaS products serving multiple independent customers must isolate data and behavior between tenants without the cost of fully separate deployments per customer. Getting tenancy wrong produces data leaks (tenant A sees tenant B's records), noisy-neighbor degradation (one tenant's bulk import slows all other tenants), and billing gaps (consumption that goes unmeasured cannot be priced). These are architectural problems that must be addressed in the data model and infrastructure layer — retrofitting tenant isolation into an existing system is expensive."
  inBriefr={

There is no shared database across organizations, no noisy-neighbor problem to solve, and no billing pipeline to instrument. Tenant isolation is achieved at the deployment level: each operator is the sole tenant of their instance. The trade-off is that operators must provision and maintain their own infrastructure rather than subscribing to a managed service.
    </>
