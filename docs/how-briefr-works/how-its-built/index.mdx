# How it's built

Track B explains how BRIEFR is engineered — schedulers, storage, API layers, and operations on one server. The primary text in each chapter describes **how BRIEFR is built**. **At enterprise scale** callout boxes are comparative only: they note how larger orgs solve the same problem without implying BRIEFR must match that scale.

## Chapters

1. [Ingestion and scheduler](/docs/how-briefr-works/how-its-built/ingestion-scheduler) — outbound feed jobs and the APScheduler owner model
2. [Resilience](/docs/how-briefr-works/how-its-built/resilience) — circuit breakers, retries, and inbound token buckets
3. [Storage](/docs/how-briefr-works/how-its-built/storage) — SQLite/PostgreSQL schema, caches, and feed tables
4. [API and auth](/docs/how-briefr-works/how-its-built/api-auth) — FastAPI routes, session auth, and admin boundaries
5. [Background jobs](/docs/how-briefr-works/how-its-built/background-jobs) — nightly correlation, OTX sync, retro-match, and maintenance crons
6. [Search and embeddings](/docs/how-briefr-works/how-its-built/search-embeddings) — local search indexes and semantic helpers
7. [Webhooks and operations](/docs/how-briefr-works/how-its-built/webhooks-ops) — alerts, backups, health checks, and multi-worker notes
