# Search and embeddings

## What it is

BRIEFR's search and embeddings layer provides three modes of retrieval for CVEs,
MITRE ATT&CK techniques, and correlation campaigns: **keyword**, **semantic**, and
**hybrid**.

Keyword search is a SQL substring and exact-match query that works on every deployment
without configuration.

Semantic search uses sentence embeddings generated by a local CPU-only ONNX model
(`BAAI/bge-small-en-v1.5`, 384 dimensions) to find results by meaning rather than exact
wording. The model runs through the `fastembed` library. Corpus embeddings (CVEs,
techniques, campaigns) are generated offline by the scheduler's backfill job. Live
semantic and hybrid search embeds the query string on the request path via
`embed_query_text()` — one short ONNX inference per non-`cve_id` query when embeddings
are enabled.

Hybrid search (the default mode) merges keyword and vector results using Reciprocal
Rank Fusion. Short queries bias toward keyword results; longer queries bias toward vector
results. When embeddings are disabled or the index is cold, hybrid falls back silently to
keyword.

The feature is **disabled by default** (`EMBEDDINGS_ENABLED=0`). Operators opt in by
setting the environment variable and ensuring the `fastembed` package is installed. The
tool is fully functional without embeddings; only the semantic and hybrid modes degrade
gracefully to keyword retrieval.

## Why we do it

Vulnerability descriptions are written by researchers, vendors, and NVD editors using
varying terminology. An analyst searching for "remote code execution in Apache HTTP
Server" may miss CVEs that describe "RCE via HTTP/2 module" or "heap overflow in
mod_proxy". Keyword search returns exact substring matches; it cannot bridge those gaps.

Sentence embeddings let BRIEFR answer natural-language queries by finding CVEs whose
descriptions have similar meaning, not just similar tokens. The same mechanism powers the
"Related CVEs" panel on a CVE detail page: given a CVE's stored vector, BRIEFR finds the
nearest neighbors without running model inference at request time.

There are two secondary use cases:

- **ATT&CK technique search (E6).** Technique names and descriptions are embedded
  alongside CVEs, so a query for "credential theft" can surface `T1003 OS Credential
  Dumping` even if those exact words are absent.
- **Campaign search (E8).** Correlation campaigns are embedded by label, adversary,
  malware family, and tags, so a query for a threat actor name or malware family name
  returns the corresponding campaign.

## Where it lives in BRIEFR

The search and embeddings system spans four layers:

```text
GET /api/search/semantic
        |
  routers/search.py          (request validation, mode routing)
        |
  services/semantic_search.py (orchestration: keyword + vector + RRF merge)
        |
  db/embeddings_search.py     (SQL: keyword queries, ANN, cosine scan, RRF helpers)
  ml/embeddings.py            (model loading, vector serialization, backfill functions)
        |
  embeddings table            (entity_type, entity_id, model, dims, embedding, content_hash)
  cve_embeddings table        (legacy BLOB, read-fallback during transition)
```

The write path (embedding generation) runs entirely inside the scheduler:

```text
scheduler.py -> run_embeddings_sync()
    |
    +-> run_embeddings_backfill()           (CVEs, ml/embeddings.py)
    +-> run_technique_embeddings_backfill() (ATT&CK, ml/embeddings.py)
    +-> run_campaign_embeddings_backfill()  (campaigns, ml/embeddings.py)
    |
    -> embeddings table (pgvector vector on Postgres; BLOB on SQLite)
```

## How it works

### The embeddings table

The primary storage target is the `embeddings` table, keyed by `(entity_type, entity_id,
model)`. It holds three entity types: `cve`, `technique`, and `campaign`. Each row
stores the embedding as raw float32 little-endian bytes (`BLOB` on SQLite, a
`vector(384)` column on PostgreSQL), plus a `content_hash` computed as
`sha256(embed_text + "\n" + model_name)`.

On **PostgreSQL**, Alembic migration `032_embeddings_pgvector` installs the `vector`
extension and creates an HNSW cosine index (`idx_embeddings_cve_hnsw`) filtered to CVE
rows. This enables approximate nearest-neighbor (ANN) search directly in SQL via the
`<=>` distance operator. The migration requires `pgvector/pgvector:pg16`; plain
`postgres:*-alpine` images lack the extension.

On **SQLite**, there is no ANN index. Cosine similarity is computed in Python using NumPy:
all stored vectors for the relevant entity type are loaded into a matrix, and a dot
product against the query vector gives cosine scores (vectors are L2-normalized at write
time, so cosine equals dot product).

The legacy `cve_embeddings` table remains as a read fallback for the related-CVE path.
New vectors are dual-written to both tables during the transition; the newer multi-entity
`embeddings` table is preferred.

### Embed text construction

Before calling the model, BRIEFR builds a deterministic text string from each entity's
fields (implemented in `db/embeddings_pgvector.py`):

| Entity | Fields concatenated |
| --- | --- |
| CVE | `description`, `summary`, `affected_products`, `cwe_ids` (up to 2 000 chars) |
| ATT&CK technique | `name`, `tactic`, `description` (up to 2 000 chars) |
| Campaign | `label`, `adversary`, `malware_families`, `tags` (up to 2 000 chars) |

The `content_hash` field encodes both the text and the model name, so a model upgrade
or a CVE description edit automatically invalidates the stored vector. The next backfill
run detects the hash mismatch and re-embeds.

### Scheduler backfill

The `embeddings_backfill` job runs every 6 hours (default; configurable via
`EMBEDDINGS_SYNC_INTERVAL_HOURS`). Its first fire is scheduled 90 seconds after backend
startup. A single lock (`asyncio.Lock` from `scheduler_locks.py`) prevents a manual
admin trigger from overlapping a scheduled run.

One job body (`run_embeddings_sync`) calls three functions in sequence:

1. `run_embeddings_backfill` — CVEs missing a vector, or with a `migrated:` placeholder
   hash, or with a content-hash mismatch. Processed newest-first; capped per run by
   `EMBEDDINGS_MAX_PER_RUN` (default 2 000).
2. `run_technique_embeddings_backfill` — ATT&CK techniques missing or stale; capped at
   800 per run.
3. `run_campaign_embeddings_backfill` — non-retracted campaigns missing or stale; capped
   at 500 per run.

Each function batches inference in groups of 64 (`EMBED_BATCH_SIZE`), commits after
each batch, and writes progress messages to `_job_progress` so the admin scheduler UI
can show live progress.

The model is loaded lazily on the first backfill run that actually needs to embed
something. Loading the ONNX weights costs roughly 100 MB RSS; the singleton is kept
alive for the process lifetime once loaded.

**Auto-on-ingest (`EMBEDDINGS_AUTO_ON_INGEST`, default `1`):** when new CVEs arrive
via the NVD or cvelistV5 ingest jobs, the ingest path queues those CVE IDs for
immediate re-embedding (up to `EMBEDDINGS_INGEST_MAX_PER_RUN`, default 25 per ingest
event). This keeps newly ingested CVEs searchable semantically without waiting for the
next 6-hour backfill cycle.

**Catch-up mode:** when an operator activates catch-up mode (an admin feature for
accelerated backfill after a gap), `EMBEDDINGS_MAX_PER_RUN` is doubled, capped at 5 000
per run.

### Request path: query classification and search modes

`GET /api/search/semantic` accepts four parameters that control what the search does:

| Parameter | Default | Effect |
| --- | --- | --- |
| `q` | `""` | The query string (max 500 chars). |
| `mode` | `"hybrid"` | `"hybrid"`, `"keyword"`, or `"semantic"`. |
| `limit` | `20` | Max results (1–50). |
| `stack`, `severity`, `kev_only` | — | Optional CVE post-filters. |

The first step is query classification (`classify_query_shape`):

| Shape | Condition | Effect |
| --- | --- | --- |
| `cve_id` | Input matches `CVE-YYYY-NNNNN` | Hybrid uses keyword-first; semantic uses stored neighbors. |
| `short` | 1–2 tokens | Hybrid weight: 70% keyword, 30% vector. |
| `long` | 3+ tokens | Hybrid weight: 40% keyword, 60% vector. |

For `cve_id` queries, behavior depends on the requested mode. In **hybrid** mode, BRIEFR
returns keyword hits only (`meta.method`: `"keyword_first"`); no vector lookup runs. In
**semantic** mode, the router calls `find_similar_via_embeddings_table` — stored vectors
for the matching CVE ID, with no live query embedding. For all other query shapes, if
embeddings are enabled, `embed_query_text()` embeds the query string once, then
`ann_search_by_query_vector` retrieves the top-k nearest neighbors from the `embeddings`
table for all three entity types.

### Hybrid merge (Reciprocal Rank Fusion)

RRF scores each candidate by summing `weight / (60 + rank)` across the keyword and
vector ranked lists, where the weight differs by query shape:

| Query shape | Keyword weight | Vector weight |
| --- | --- | --- |
| `short` | 0.7 | 0.3 |
| `long` | 0.4 | 0.6 |

The merged CVE list fills a budget slot first, with separate smaller budgets reserved for
ATT&CK techniques and campaigns so typed hits are not crowded out by CVEs. Stack,
severity, and KEV filters are applied as a post-filter pass on the CVE candidate pool
before hydration.

The `meta` object in every response reports the path used: `"hybrid"`, `"keyword"`,
`"semantic"`, `"keyword_first"` (CVE-id hybrid), or `"keyword_fallback"` (vector empty
or disabled).

### Search API tokens (E5)

The search endpoint can be called with either a session cookie (browser) or a dedicated
search API token. Tokens are created by an admin, stored bcrypt-hashed in the
`search_api_tokens` table (Alembic migration `033_search_api_tokens`), and issued once
in plaintext (prefix `briefr_search_`). Verification uses a 26-character lookup prefix
for fast indexed lookup, then bcrypt comparison. Tokens carry scopes; the default scope
set is `["search:semantic", "cves:related", "cves:read"]`.

### LLM product extraction (optional)

A separate scheduler job (`LLM_PRODUCT_EXTRACTION_ENABLED=1`) uses a configured LLM
provider to extract `{vendor, product, version_range}` from CVE descriptions that NVD
has not yet analyzed (empty `affected_products`). Extracted products are written back as
`affected_products_source='llm'`. Official CPE data from NVD supersedes LLM output on
the next NVD upsert. This is a keyword-side enrichment: it improves stack-match quality
and the embed text for CVEs that lack CPE data, but it is not part of the vector search
path itself. Completed extractions (including empty ones) are negative-cached in
`feed_cache` under `llm_products:<cve_id>` for 168 hours to avoid re-billing the LLM
provider. A crash-safe staging mechanism stores the raw LLM response before parsing so a
restart replays the response instead of making a second API call.

## What it needs

Embeddings are off by default. To enable them, set:

| Setting | Description |
| --- | --- |
| `EMBEDDINGS_ENABLED` | Set `1` to enable model inference and semantic search. Default: `0`. |
| `fastembed` package | Must be installed: `pip install fastembed`. Not in the base requirements; the backend starts without it and logs a warning when `EMBEDDINGS_ENABLED=1` but the package is absent. |
| `EMBEDDINGS_MODEL` | Embedding model name. Default: `BAAI/bge-small-en-v1.5` (384-dim, ~130 MB ONNX weights). |
| `EMBEDDINGS_CACHE_DIR` | Directory where model weights are downloaded and cached. Default: fastembed's default (`~/.cache/fastembed`). In production (systemd `ProtectSystem=strict`), set this to a writable path such as `/var/lib/briefr/models`. |
| `EMBEDDINGS_MAX_PER_RUN` | CVEs embedded per scheduled backfill run. Default: `2000`. |
| `EMBEDDINGS_INGEST_MAX_PER_RUN` | CVEs re-embedded per ingest event (auto-on-ingest). Default: `25`. |
| `EMBEDDINGS_AUTO_ON_INGEST` | Re-embed newly ingested CVEs immediately. Default: `1` when `EMBEDDINGS_ENABLED=1`. |
| `EMBEDDINGS_SYNC_INTERVAL_HOURS` | Backfill cadence. Default: `6`. |
| `EMBEDDINGS_PGVECTOR` | Set `0` to skip writes to the `embeddings` table (legacy-only). Default: `1`. |
| pgvector image | PostgreSQL must use `pgvector/pgvector:pg16`. Alembic migration `032` fails on plain Postgres images that lack the `vector` extension. |

LLM product extraction (separate, optional):

| Setting | Description |
| --- | --- |
| `LLM_PRODUCT_EXTRACTION_ENABLED` | Set `1` to enable. Requires at least one LLM provider API key. Default: `0`. |
| `LLM_PRODUCT_EXTRACTION_MAX_PER_RUN` | CVEs processed per scheduler run. Default: `10`. |

The admin UI exposes an embeddings health panel (backed by
`services/retrieval_health.py`) that reports:

- Whether embeddings are enabled and which model is configured.
- Row counts in `embeddings` by entity type.
- Pending counts (CVEs, techniques, campaigns missing vectors or with migrated
  placeholders).
- Last backfill run timestamp, records upserted, and error status.
- Whether the pgvector extension is present (Postgres only).
- A `degraded` indicator with a reason: `"disabled"`, `"no_vector_extension"`, or
  `"cold_index"`.

## How industry does it — and why BRIEFR does it this way

<AtEnterpriseScale>
  <div>
    Large vulnerability management platforms typically run embeddings in a dedicated
    vector database (Pinecone, Weaviate, Qdrant, or a managed pgvector service), generated
    by a GPU-accelerated inference service or a cloud embedding API. The embedding pipeline
    is a separate microservice that consumes events from a message queue. The vector index
    is queried independently from the relational database. Search tokens and rate limiting
    are handled by an API gateway layer.
  </div>

  <div>
    BRIEFR keeps embeddings inside the same database as the rest of the corpus.
    On Postgres, pgvector runs in the same container and the HNSW index is a regular SQL
    index on the <code>embeddings</code> table. On SQLite, cosine similarity is computed
    in Python over loaded BLOB bytes. The ONNX model runs on CPU via fastembed — no GPU
    is required. There is no separate inference service, message queue, or external vector
    store to operate. Embedding generation is a scheduler job in the same process as
    ingestion and the API.
  </div>
</AtEnterpriseScale>

The trade-off is familiar: operational simplicity against scale. A self-hosted BRIEFR
instance gets semantic search and related-CVE recommendations with no additional
infrastructure. The CPU-only ONNX model is slower than a GPU-backed service, which is why
bulk inference is scheduler-side and request-path search only embeds a single short query
string. On a catalog of tens of thousands of CVEs the batch catches up within hours; on a
significantly larger corpus the backfill window would grow proportionally.

The pgvector HNSW index makes ANN query time sub-millisecond on Postgres even at scale,
so the bottleneck is always the write path (embedding generation), not the read path
(ANN retrieval). Operators who outgrow the scheduler-based backfill rate can raise
`EMBEDDINGS_MAX_PER_RUN`, activate catch-up mode, or run the backfill job more
frequently via `EMBEDDINGS_SYNC_INTERVAL_HOURS`.
